# FragSense Repository Agent Instructions

## Mission

You are the implementation agent for **FragSense**, a runtime fragmentation-aware Zone reclaim and layout-reorganization extension for RocksDB-ZenFS on ZNS SSDs.

The immediate goal is not to implement every paper feature at once. Work milestone by milestone, preserve Native ZenFS behavior by default, and produce testable commits with explicit evidence.

The authoritative implementation plan is:

- `docs/fragsense/IMPLEMENTATION_PLAN.md`
- `docs/fragsense/CODEX_RUNBOOK.md`

Read both before changing code.

## Non-negotiable correctness rules

1. **Native ZenFS remains the default.**
   - FragSense must be disabled unless explicitly enabled by configuration.
   - With FragSense disabled, file placement, metadata persistence, recovery, reset behavior, and performance-critical paths must remain semantically unchanged.

2. **Do not misrepresent Native ZenFS.**
   - Native ZenFS uses Lazy Reset; do not describe it as an active victim-selection GC.
   - FragSense adds an optional host-side active reclaim/reorganization path outside Lazy Reset.

3. **Never fabricate results.**
   - Do not hard-code or claim the paper’s percentage improvements.
   - Only report measurements generated by real runs and archived raw logs.

4. **No destructive device operations without explicit human confirmation.**
   - Do not run `mkfs`, Zone Reset, device formatting, `nvme zns reset-zone`, or writes to `/dev/nvme*` automatically.
   - Provide the exact command and wait for the user to run or approve it.
   - Unit/integration tests should use mocks, fake zoned backends, nullblk, FEMU, or a user-approved device.

5. **Crash consistency is mandatory for active migration.**
   - Use copy-before-switch semantics.
   - Never reset a source Zone until the new extent data and new mapping are durable and the source has no live references.
   - Every metadata transition must have a recovery story and tests.

6. **Keep measurement semantics explicit.**
   - Distinguish extent-level fragmentation from block-level fragmentation.
   - `Hot(z)` is a lightweight recent foreground-update signal, not a precise lifetime predictor.
   - Compaction outputs must not silently contaminate the foreground Hot(z) window.

7. **One milestone per task.**
   - Do not implement future milestones unless the current milestone’s acceptance criteria pass.
   - Do not combine state sensing, migration, elastic control, and layout regrouping in one unreviewable patch.

## Development workflow

1. Inspect the repository and existing `AGENTS.md`, build system, tests, ZenFS version, RocksDB version, and current observer patches.
2. Before editing, write a short implementation note containing:
   - files and symbols to change;
   - invariants at risk;
   - tests to add;
   - rollback/feature-flag strategy.
3. Prefer small commits. Suggested branch/commit sequence:
   - `fragsense/m0-audit`
   - `fragsense/m1-state`
   - `fragsense/m2-dry-run`
   - `fragsense/m3-evacuation`
   - `fragsense/m4-policies`
   - `fragsense/m5-elastic-io`
   - `fragsense/m6-regrouping`
4. Run formatting, compilation, and the narrowest relevant tests after every logical change.
5. At the end of each task, report:
   - changed files;
   - behavior added;
   - tests run and their exact commands;
   - known limitations;
   - whether Native mode was regression-tested;
   - next milestone, without starting it.

## Coding constraints

- Follow the repository’s existing C++ standard, naming, status/error handling, logging, and synchronization conventions.
- Avoid new global mutable state.
- Use RAII for locks, files, buffers, and Zone handles.
- Avoid blocking foreground RocksDB paths with whole-Zone scans or optimization solvers.
- Prefer incremental counters and interval-local updates.
- Protect Zone statistics and mappings with the smallest correct lock scope.
- Add structured metrics rather than parsing free-form logs when practical.
- New configuration options must have documented defaults and validation.

## Required feature flags

Names may be adapted to existing configuration conventions, but equivalent controls must exist:

- `fragsense_enabled=false`
- `fragsense_dry_run=true`
- `fragsense_policy=native|reclaimable|valid_ratio|frag|cff|cff_filter|cff_elastic|full`
- `fragsense_block_size`
- `fragsense_window_events`
- `fragsense_reclaim_threshold`
- `fragsense_min_age`
- `fragsense_budget_base_bytes`
- `fragsense_budget_min_bytes`
- `fragsense_budget_max_bytes`
- `fragsense_frag_target`
- `fragsense_control_period_ms`
- `fragsense_bandwidth_bytes_per_sec`
- `fragsense_compaction_pressure_threshold`

All active behavior must be off by default.

## Required observability

Produce structured records sufficient for paper experiments. At minimum include:

- timestamp/control cycle;
- Zone ID/state/age;
- written/live/invalid bytes;
- reclaimable ratio `R(z)`;
- valid block count and live segment count;
- `Frag(z)` and global `Frag_current(t)`;
- `Hot(z)`/`Cold(z)`;
- candidate admission result and reject reason;
- policy score, cost, density, rank, and selected flag;
- migration source/target bytes and duration;
- mapping switch and reset status;
- free/open/active/blocked Zone counts;
- budget raw/smoothed/clamped/consumed;
- compaction pressure and throttle/pause state.

Do not log user keys or values.

## Test expectations

- State accounting tests for allocation, invalidation, deletion, reset, and recovery rebuild.
- Segment-count tests for boundary invalidation and interval splitting/merging.
- Sliding-window tests for insertion, eviction, and foreground/compaction separation.
- Candidate/filter/score tests including zero-live-byte and newly sealed Zones.
- Native-disabled regression tests.
- Dry-run tests proving no migration or reset occurs.
- Migration fault-injection tests before copy completion, before mapping switch, after switch, and before reset.
- Recovery tests proving a file resolves to exactly one durable extent mapping.
- Concurrency tests around reads, file deletion, migration, and reset.

## Stop conditions

Stop and ask for review when:

- the existing ZenFS metadata format cannot represent a safe extent relocation;
- distinguishing foreground flush output from compaction output requires an invasive RocksDB API change;
- a test would require destructive access to a real device;
- an implementation choice conflicts with the current paper semantics;
- Native mode changes unexpectedly;
- source reset safety cannot be proved.
